# 文档目录规则

`docs/` 是项目知识库，不是临时草稿堆。Codex 在这里新增或修改内容时，必须保证可追溯、可复核、可复用于答辩。

## 1. 事实源与引用

- 涉及器件参数、MCSDK/CubeMX 配置、比赛规则、OpenAI/Codex 能力边界时，必须优先查官方或高可信来源，并在文档中写清来源。
- 涉及项目当前方案时，优先服从 `docs/00_project_truth/project_context.md`、V9 最终版、技术报告 V1.0 和实测数据。
- 不确定的内容写 `TODO`，不要为了让文档显得完整而编造参数、结论或测试结果。

## 2. 版本归档

- V7/V8/V7.1 旧申报书只能用于追溯方案演进，不能覆盖 V9 的当前事实。
- 旧版本分析、废弃方案、历史盲区复盘放入 `docs/90_archive/` 或 `materials/`，不要混进当前执行方案。

## 3. 答辩材料约束

- `docs/06_defense_pack/` 中的每个卖点都要能对应到三件事：数据、来源、演示锚点。
- 没有实测数据的效果描述只能写成待验证假设，不能写成已经实现。
- 答辩话术要能回到项目事实源、实验记录或协议文档，避免只讲概念。

## 4. 实验与日志

- 联调、异常定位、波形记录、串口日志必须优先落到 `experiments/YYYY-MM-DD_topic/`。
- 文档中引用实验结论时，要写明实验日期、设备状态、输入条件和观察到的现象。
